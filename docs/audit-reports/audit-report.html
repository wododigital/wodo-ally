Executive Summary

WODO Ally is an ambitious internal financial management platform built with Next.js 14, Supabase, and React PDF. The application demonstrates strong foundational architecture - the tech stack choices are sound, the database schema is thoughtfully designed, the UI design language is visually polished with a distinctive dark glass-card aesthetic, and the codebase follows modern React patterns.

However, the platform has critical defects across financial calculations, security, and core workflows that prevent it from being trusted with real money or investor reporting in its current state. The most severe issues include invoice PDFs directing clients to the wrong bank account, floating-point arithmetic without rounding, and quarterly investor reports covering only 2 months instead of 3.

The good news: most issues are fixable with targeted effort. The architecture is clean enough that corrections can be made without major refactoring. With approximately 3-4 weeks of focused remediation, this platform could reach a state suitable for production use and investor reporting.

Financial Integrity Assessment

Score: 3/10 - The financial logic has critical defects that will produce incorrect outcomes on real transactions.

Critical Invoice Calculations

No rounding on any calculation. All three paths (new invoice modal, edit page, retainer auto-draft) use raw floating-point arithmetic. The DB column NUMERIC(12,2) truncates (not rounds), creating systematic under-billing. Example: tax of 17999.9982 stored as 17999.99, not 18000.00.

Edit page does not update line items. updateInvoice.mutate() only updates the invoices table. The invoice_line_items table retains old values. After edit: PDF shows old items, totals show new values.

Files: new-invoice-modal.tsx:85-87, edit/page.tsx:117-120, use-invoices.ts:407-411

Critical Payment & Balance Logic

TDS not deducted from balance_due. Formula: balanceDue = Math.max(0, total_amount - totalReceived). TDS tracked separately but never subtracted. Every TDS-affected invoice stuck as "partially_paid".

Floating-point accumulation. .reduce() on amount_received uses raw addition. Three payments of Rs.33.33 sum to 99.99000000000001.

Overpayments silently accepted. No validation checks amount_received <= balance_due. Rs.999,999 on a Rs.17,500 invoice accepted.

No overdue auto-detection. No cron, trigger, or on-access check. Overdue KPI perpetually undercounts.

Files: use-invoices.ts:834-841, 811-891, 852

Critical Tax Handling (GST/IGST/TDS)

GST split not implemented. Indian tax law requires CGST (9%) + SGST (9%) for intra-state, IGST (18%) for inter-state. System shows single "GST @ 18%" for all.

GSTIN inconsistency. Email templates: 29AADCW8591N1ZA. Settings page: 29AAECW2882M1ZW. Different GSTINs in official communications is a compliance risk.

High Multi-Currency Issues

Indian numbering on all currencies in PDF. Uses toLocaleString("en-IN") for USD/GBP. $1,234,567 displays as "$12,34,567".

EUR falls through to Rs. No EUR case in switch - Euro amounts show "Rs." prefix.

Revenue mixing currencies. Dashboard uses amount_received_inr ?? amount_received. A $5,000 payment without INR conversion counted as Rs.5,000.

Critical Revenue Recognition & P&L

Pro Forma conversion invisible. Marked "paid" but no invoice_payments record created. Zero contribution to revenue metrics.

Transfers/loans counted as expenses. monthly_pl_view filters WHERE transaction_type != 'income' but still includes transfer, tax, loan. P&L overstates expenses.

Client health scores double-counted. JOIN between invoices and payments produces one row per payment - 3 partial payments = invoice counted 3 times.

Quarterly report: 2 months instead of 3. use-reports.ts:110 - quarterEnd uses month+1 instead of month+2. Every quarterly report understates by ~33%.

Email System

Readiness: 15% - Invoice email flow is completely broken (stub endpoint).

The invoice email flow is non-functional. The UI calls /api/invoices/[id]/send-email which is a stub containing a TODO comment. It returns { success: true } without sending. Users see "Email sent successfully" but nothing is dispatched. Only the investor report email flow is properly wired.

Before Activating SMTP

  • Wire invoice email modal to /api/email/send instead of stub endpoint
  • Add PDF attachment generation for invoice emails
  • HTML-escape all template variables (XSS risk)
  • Remove tls: { ciphers: "SSLv3" } from SMTP config
  • Add SMTP credential detection with clear error messaging
  • Create Nodemailer transport singleton (currently recreated per email)
  • Add retry logic for transient SMTP failures
  • Replace display: flex with table layouts for Outlook compatibility
  • Update invoice status to "sent" after successful delivery
  • Add email format validation
  • Connect Settings template editor to send flow or remove it

What Works Well

Despite critical issues, WODO Ally demonstrates strong fundamentals that provide a solid base for remediation.
Thoughtful Database Design

Proper data types (NUMERIC(12,2) for money, UUID for IDs), appropriate FK relationships, RLS policies on all major tables. Migration files well-organized.

Consistent Hook Architecture

All data access through TanStack Query with uniform error handling. Clean separation of concerns across hooks.

Professional PDF Structure

Correct Indian FY Handling

April-March boundaries, quarter definitions, FY string formatting all correctly implemented throughout the codebase.

Strong Auth Foundation

All 14 routes protected. API routes check auth. Supabase client/server/admin separation correctly implemented.

Distinctive Visual Design

Comprehensive Feature Scope

Clean Code Quality

No console.log in production. No commented-out code. No em dashes. TypeScript strict mode. ESLint passing.
